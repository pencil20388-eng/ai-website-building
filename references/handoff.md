# 交付给开发

AI 做出来的是一个原型，最终要交给开发团队接入到正式网站。这一步没做好，前面白做。

## 先想清楚：原型 ≠ 最终上线版

交付前必须和开发对齐一件事：AI 生成的 html 是**原型**，研发需要把它转成正式网站用的技术（不管是什么框架），不是复制粘贴就能上线。这个预期不对齐，后面会扯皮。

## 交付物清单

给开发的不只是一个文件，而是一整包：

1. **网页文件**（index.html）— 完整能打开的原型。
2. **图片资源包** — 所有用到的图片，按模块命名好（比如 `hero-认证-soc2.png`），别给一堆 `1.png 2.png`。
3. **改动清单** — 列出哪些是新改的、哪些保持原样，开发据此估工时。
4. **链接清单** — 每个按钮、每个链接点击后跳去哪，能填的先填好，填不了的标注「待补充」。
5. **设计规范**（可选）— 把间距/颜色/字号规范一起给，方便开发还原。

## 哪些「照着抄」、哪些「开发做」

帮开发分清楚边界，这是他们最关心的（直接影响工时估算）：

- **照着抄**（直接用原型的 HTML/CSS）：静态展示部分、布局、样式。
- **开发自己做**：导航下拉菜单等交互逻辑、需要从后台读取的内容、图片换成正式 CDN 地址、字体文件确认、老旧浏览器的兼容降级、移动端最终适配确认。
- **复用现有的**：如果只是改首页，导航栏和页脚通常复用现有网站的，不用重做，原型里的只是占位参考。

## 需求文档模板

给开发提需求，用这个结构，清楚又省沟通：

```
# 需求：XXX 模块

## 目标
一句话说清楚要做什么

## 当前状态
截图 / 链接 / 当前版本

## 修改内容
1. 具体改动一
2. 具体改动二

## 验收标准
- [ ] 改完刷新能看到
- [ ] 手机端显示正常
- [ ] 线上验证通过

## 优先级 / 截止时间
P0 / P1 / P2，几月几号前
```

## 提前准备的资源（否则会卡开发）

| 资源 | 说明 | 提前量 |
|------|------|--------|
| Logo 文件 | 透明背景 PNG 或 SVG | 1 天 |
| 认证徽章/图标 | 官方提供，注意版权 | 2 天 |
| 文案 | 标题、描述、按钮文字 | 1 天 |
| 字体文件 | 确认可商用 | 3 天 |
| 各模块配图 | 正式版图片 | 按需 |
| 各按钮跳转链接 | 注册、下载、联系客服等 | 1 天 |

## 排期沟通

- 先做技术评审再排期：让开发先看一遍原型，提出技术风险，再基于评审估工时，别让他们盲排。
- 资源和需求同时准备，不要串行等。
- 明确移动端适配的责任方和验收标准，别让「手机端谁负责」变成模糊地带。

## 埋点（数据分析）必须一起提

如果改版后要评估效果（看转化有没有提升、用户在哪流失），埋点必须在开发评审时一起提，和页面同步开发。否则极容易「页面上线了、埋点没做」，改版数据无法回溯，等于白改。

埋点需求要说清楚：
- 要追踪哪些行为（页面访问、各模块曝光、各按钮点击、注册完成等）
- 每个事件在页面的什么位置触发
- 要能按来源、设备、版本拆分数据
- 部署 deadline：建议在页面上线前完成，确保上线即有数据
